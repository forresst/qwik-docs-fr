En plus de la réactivité implicite créée par les templates, Qwik prend en charge l'exécution explicite de code quand une propriété change. Ceci est réalisé par le hook [`useWatch$()`](/docs/components/lifecycle/index.mdx#usewatch). Les hooks [`useWatch$()`](/docs/components/lifecycle/index.mdx#usewatch) s'exécutent avant le rendu du composant et peuvent être asynchrones. Le hook peut également avoir une fonction de nettoyage qui est invoquée lors de la prochaine exécution du hook ou lorsque le composant est supprimé.

Dans cet exemple, cliquer sur `+1` met à jour le `count` immédiatement. Ce que nous voulons, c'est mettre à jour le `delay count` après un délai de 2 secondes. Si `count` est mis à jour avant que les 2 secondes soient écoulées, alors le temporisateur est redémarré.

Remarquez que le callback [`useWatch$()`](/docs/components/lifecycle/index.mdx#usewatch) reçoit la fonction `track`. Utilisez la fonction `track` pour dire à Qwik quelles propriétés doivent déclencher cette surveillance. La fonction `track` crée des abonnements dans le store. A chaque invocation de [`useWatch$()`](/docs/components/lifecycle/index.mdx#usewatch) les abonnements sont effacés, donc il est important de toujours mettre en place un nouveau jeu d'abonnements. Ceci est utile si le jeu d'abonnements change pendant la durée de vie de la fonction.

La fonction callback [`useWatch$()`](/docs/components/lifecycle/index.mdx#usewatch) peut retourner une fonction de _cleanup_ (NdT « nettoyage »). La fonction de nettoyage est appelée lors de la prochaine exécution de la callback [`useWatch$()`](/docs/components/lifecycle/index.mdx#usewatch) ou lorsque le composant est supprimé. Dans notre cas, la fonction de nettoyage est utilisée pour retourner le code qui efface le `setTimeout`.

Les callbacks [`useWatch$()`](/docs/components/lifecycle/index.mdx#usewatch) s'exécutent avant le rendu du composant. Cela leur permet d'être utilisés pour calculer les valeurs utilisées dans le rendu. La fonction s'exécute à la fois sur le serveur et sur le client. L'exécution sur le serveur met en place les abonnements qui sont ensuite sérialisés et disponibles pour le client. Cela évite au client de devoir charger tous les composants et de les exécuter au moins une fois pour récupérer les informations d'abonnement pour le système.
