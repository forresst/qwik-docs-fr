# État

La gestion de l'état est une partie importante de toute appli. Dans Qwik, nous pouvons différencier deux types d'état, réactif et statique.

L'état statique est tout ce qui peut être sérialisé : une chaîne, un nombre, un objet, un tableau... n'importe quoi. L'état réactif, quant à lui, est créé avec `useStore()`.

Il est important de noter que l'état dans Qwik n'est pas un état de composant, mais plutôt un état d'appli qui peut être instancié par n'importe quel composant.

## `useSignal()`

`const signal = useSignal(initialState)` est un hook qui crée un signal réactif. Il prend une valeur initiale et renvoie un signal réactif.

Le signal réactif renvoyé par `useSignal()` consiste en un objet avec une seule propriété `signal.value`. Si vous modifiez une propriété de valeur de l'objet, tout composant qui en dépend sera mis à jour.


### Exemple

Un exemple montrant comment `useSignal` est utilisé dans l'exemple Counter pour garder la trace du compteur.

```tsx
export default component$(() => {
  const count = useSignal(0);

  return (
    <>
      <button onClick$={() => count.value++}>Incrémenter</button>
      Compteur : {count.value}
    </>
  );
});

```

Dans l'exemple ci-dessus, le composant App utilise `useSignal()` pour créer un signal. Le signal est ensuite utilisé pour suivre le compteur.

En accédant simplement à la propriété `count.value`, le composant sera mis à jour si la valeur du signal change, lorsque la propriété est modifiée dans le gestionnaire de clic.


## `useStore()`

Fonctionne de manière très similaire à `useSignal()`, mais prend un objet comme valeur initiale, pensez à un store comme un signal à valeurs multiples, ou un objet composé de plusieurs signaux.


`const reactive = useStore(initialState)` est un hook qui crée un objet réactif. Il prend un état initial comme paramètre, et retourne un objet réactif.

L'objet réactif renvoyé par `useStore()` est comme n'importe quel autre objet, mais il est réactif. Si vous changez une propriété de l'objet, tout composant qui en dépend sera mis à jour.

> **REMARQUE**
> Pour que la réactivité fonctionne, assurez-vous de garder une référence sur l'objet réactif et pas seulement sur ses propriétés. Par exemple, faites `let { count } = useStore({ count : 0 })` et ensuite la mutation de `count` ne déclenchera pas la mise à jour des composants qui en dépendent.

### Exemple

Un exemple montrant comment `useStore` est utilisé dans l'exemple Counter pour garder le suivi du compteur.

```tsx
export default component$(() => {
  const state = useStore({ count: 0 });

  return (
    <>
      <button onClick$={() => state.count++}>Increment</button>
      Compteur: {state.count}
    </>
  );
});

```

Dans l'exemple ci-dessus, le composant App utilise `useStore()` pour créer un objet réactif. Cet objet est ensuite utilisé pour garder le suivi du compteur. Le compteur est ensuite affiché dans le composant.

En accédant simplement à la propriété `state.count`, le composant sera mis à jour si le compteur change, lorsque la propriété est modifiée dans le gestionnaire du clic.

### Valeurs récursives

Par défaut, `useStore()` ne traque que les champs de premier niveau de votre store, ce qui signifie que pour que toute mise à jour soit enregistrée, vous devez mettre à jour les valeurs du champ de premier niveau.

Par exemple, ceci ne fonctionnera pas :

```tsx
import { component$, useStore } from '@builder.io/qwik';

export default component$(() => {
  const store = useStore({
    nested: { fields: { are: "not tracked" }}
  })

  return (
    <>
      <p>{store.nested.fields.are}</p>
      <button onClick$={() => store.nested.fields.are = "tracked"}>Click me</button>
    </>
  );
});
```

Pour mettre à jour le suivi par défaut, nous devons mettre à jour le champ "nested" de niveau supérieur :

```ts
store.nested = { fields: { are: { "tracked" } } }
```

Pour que le premier exemple fonctionne, nous pouvons passer un deuxième argument à notre useStore, et lui demander d'utiliser la récursion pour suivre toutes les valeurs de notre magasin, quelle que soit leur profondeur.

```tsx
export default component$(() => {
  const store = useStore({
    nested: { fields: { are: "not tracked" }}
  }, { recursive: true })

  return (
    <>
      <p>{store.nested.fields.are}</p>
      <button onClick$={() => store.nested.fields.are = "tracked"}>Click me</button>
    </>
  );
});
```

Maintenant, le composant sera mis à jour comme prévu. Cela permettra également de suivre les valeurs individuelles dans les tableaux !

```tsx
import { component$, useStore } from '@builder.io/qwik';

export default component$(() => {
  const store = useStore({
    letters: ["A", "B", "C"]
  }, { recursive: true })

  return (
    <>
      {store.letters.map(letter => <p>{letter}</p>)}
      <button onClick$={() => { store.letters[2] = "Z"}}>Click me</button>
    </>
  );
});
```


## Transmission d'un store à d'autres composants

Une des fonctionnalités intéressantes de Qwik est que l'état peut être passé à d'autres composants, et ils peuvent à la fois lire et écrire dessus, permettant aux données de circuler à travers l'arborescence dans toutes les directions.

Il y a trois façons de passer l'état à d'autres composants.

### Utilisation des props

La façon la plus naïve de passer l'état à d'autres composants est de le passer comme props. C'est la façon dont vous le feriez dans React, et cela fonctionne aussi dans Qwik.

```tsx
export const Parent = component$(() => {
  const userData = useStore({
    count: 0,
  });

  return (
    <>
      <Child userData={userData} />
    </>
  );
});

export const Child = component$(({ userData }) => {
  return (
    <>
      <button onClick$={() => userData.count++}>Increment</button>
      Compteur : {userData.count}
    </>
  );
});
```

### Utilisation de l'API de contexte

L'API de contexte est un moyen pour transmettre un état aux composants sans avoir à le transmettre à travers les props. Automatiquement, tous les composants descendants de l'arborescence peuvent obtenir une référence à l'état, et avoir l'accès en lecture/écriture.

Consultez l'[API de contexte](../context/index.mdx) pour plus d'informations.

```tsx
const CTX = createContext('stuff');

export const Stores = component$(() => {
  const userData = useStore({
    count: 0,
  });

  useContextProvider(CTX, userData);

  return (
    <>
      <Child />
    </>
  );
});

export const Child = component$(() => {
  const userData = useContext(CTX);
  return (
    <>
      <button onClick$={() => userData.count++}>Increment</button>
      Compteur : {userData.count}
    </>
  );
});
```

## Valeurs calculées

Les valeurs calculées sont des valeurs qui sont dérivées d'autres valeurs. Elles sont utiles lorsque vous avez une valeur qui est dérivée d'autres valeurs, et que vous voulez la maintenir en synchronisation avec les autres valeurs.

Dans Qwik il y a deux façons de créer des valeurs calculées, en utilisant `useWatch$()` ou `useResource$()`.

La principale différence entre les deux est que `useWatch$()` permet les effets de bord et l'exécution est sérialisée, tandis que `useResource$()` est asynchrone et plusieurs appels useResource peuvent se produire en parallèle.

`useWatch$()` est généralement utilisé pour calculer l'état intermédiaire, alors que `useResource$()` a une meilleure ergonomie pour calculer l'état final, utilisé pour le rendu. Voyons quelques exemples :


### Exemple de `useWatch$()`

```tsx
export default component$(() => {
  const state = useStore({
    count: 0,
    doubleCount: 0
  });

  useWatch$(({ track }) => {
    track(() => store.count);
    state.doubleCount = state.count * 2;
  });

  return (
    <>
      <button onClick$={() => state.count++}>Increment</button>
      Compteur : {userData.count}
      Compteur * 2 : {state.doubleCount}
    </>
  );
});
```

### Exemple de `useResource$()`

```tsx
export default component$(() => {
  const state = useStore({
    count: 0,
  });

  const doubleCount = useResource$(({ track }) => {
    track(() => store.count);
    return state.count * 2;
  });

  return (
    <>
      <button onClick$={() => state.count++}>Increment</button>
      Compteur : {userData.count}
      Compteur * 2 : {doubleCount.promise}
    </>
  );
});
```

Les notions de [useWatch](../lifecycle/index.mdx#usewatch) et de [useResource](../resource/index.mdx) sont expliquées en détail dans leurs sections respectives.

## Réactivité

Grâce à la réactivité fine de Qwik, seuls les composants qui dépendent de l'état seront mis à jour. C'est un énorme gain de performance, car seuls les composants qui ont besoin d'être mis à jour le seront.


## `noSerialize()`

Parfois, vous souhaitez stocker une valeur dans l'état, mais vous ne voulez pas qu'elle soit sérialisée. Ceci est utile pour stocker des valeurs qui ne sont pas sérialisables, comme les fonctions ou les classes.

```tsx
import {component$, useStore, noSerialize} from '@builder.io/qwik';

export const App = component$(() => {
  const state = useStore({
    monacoInstance: null
  });

  useClientEffect$(() => {
    // Monaco n'est pas sérialisable, donc nous ne pouvons pas le stocker dans l'état en utilisant `noSerialize()`
    state.monacoInstance = noSerialize(new Monaco());
  });
  return (
    <>
    </>
  );
});
```
